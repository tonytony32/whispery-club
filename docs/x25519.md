# X25519 — Key Agreement in Whispery

## What is X25519?

X25519 is a **Diffie-Hellman key agreement algorithm** built on Curve25519, an elliptic curve designed by Daniel J. Bernstein in 2005.

Its single purpose: allow two parties to derive the same shared secret over a public channel, without ever sending that secret to each other.

---

## The Problem It Solves

Alice and Bob want to encrypt their communication, but they can only talk over a channel where an attacker is listening. They cannot send a key directly — the attacker would intercept it.

Diffie-Hellman solves this. Think of it as mixing paint: combining two colors is easy, separating them back is practically impossible.

```
Alice has:  sk_A (private)  →  pk_A = sk_A · G   (public, she publishes it)
Bob has:    sk_B (private)  →  pk_B = sk_B · G   (public, he publishes it)

Alice computes:  sk_A · pk_B  =  sk_A · sk_B · G
Bob computes:    sk_B · pk_A  =  sk_B · sk_A · G
                                 ↑ same result
```

Both arrive at the same shared secret. An attacker sees `pk_A` and `pk_B` but cannot compute `sk_A · sk_B · G` without knowing one of the private keys.

The "X" in X25519 means only the X coordinate of the elliptic curve point is used — this simplifies the implementation and eliminates entire classes of bugs.

---

## Why Curve25519 and Not Classical DH?

Classical Diffie-Hellman operates over integers (modular arithmetic). Curve25519 operates over elliptic curve points, which gives:

| | Classical DH | X25519 |
|---|---|---|
| Secure key size | 2048–4096 bits | **256 bits** |
| Speed | slow | very fast |
| Timing attack resistance | weak | by design |
| Weak keys | possible | none — all keys are valid |
| Designed for | general use | key agreement specifically |

---

## Why Not Use the Ethereum Key Directly?

Ethereum uses **secp256k1**, a curve designed for **digital signatures** (ECDSA). Its fundamental operation is:

```
signature = sign(message, sk)
valid     = verify(signature, message, pk)
```

You can prove authorship. But secp256k1 was not designed for secure key agreement:

- It lacks the safety properties Curve25519 guarantees for DH
- It is vulnerable to timing attacks without extreme implementation care
- There are no widely audited, standard DH implementations over secp256k1
- Key operations are more expensive

The correct pattern is to use each primitive for what it was designed for:

```
secp256k1  →  identity, signatures, non-repudiation    (what Ethereum does)
X25519     →  key agreement, encryption                (what the channel does)
```

## How Whispery Uses X25519

### Deriving the X25519 keypair from an Ethereum identity

Rather than managing a separate seed, Whispery derives the X25519 keypair deterministically from the wallet's Ethereum private key using a SIWE signature as a one-way bridge:

```
eth_privkey
    │
    │  sign(SIWE message)        ← secp256k1, proves ownership
    ▼
siwe_signature
    │
    │  sha256(signature)         ← one-way, collapses 64 bytes to 32
    ▼
x25519_seed  ──▶  (x25519_sk, x25519_pk)
```

The SIWE message uses a fixed nonce (`whispery-v0-deterministic`), making the result identical every time the same wallet signs in. The Ethereum private key is never exposed to DH operations.

### Scenario A — P2P Channel

Two wallets derive a shared secret and use it to encrypt messages:

```
shared_secret = DH(sk_A, pk_B)
             = DH(sk_B, pk_A)   ← symmetric, same result from both sides

channel_id    = sha256(pk_A || pk_B)   ← sorted, deterministic
msg_key       = HKDF(shared_secret, "whispery/p2p/msg/{channel_id}/epoch/{epoch}")
ciphertext    = XSalsa20-Poly1305(message, nonce, msg_key)
```

Only A and B can compute `shared_secret`. The `channel_id` is the same regardless of who calculates it first.

### Scenario B — NFT-Gated Group Channel

The group has a dedicated X25519 keypair (`sk_group`, `pk_group`). The admin uses it to build an Access Control Table (ACT) — one entry per authorized member:

```
For each authorized member M:

    session_key           = DH(sk_group, pk_M)
    lookup_key            = HKDF(session_key, "whispery/act/lookup/{channel_id}")
    access_kdk            = HKDF(session_key, "whispery/act/access/{channel_id}")
    encrypted_content_key = secretbox(content_key, nonce, access_kdk)
```

The ACT is stored in the EEE (the channel state file). To access the channel, a member recomputes their `lookup_key` using `DH(sk_M, pk_group)` — mathematically symmetric to what the admin computed — and searches the ACT. If found, they decrypt `content_key` with their `access_kdk`.

A wallet not in the ACT (e.g. Wallet D) computes a `lookup_key` that matches no entry. Access denied, no error exposed.

```
Authorized member:     lookup_key found → decrypt content_key → read messages
Unauthorized member:   lookup_key not found → null → access denied
```

### Key Rotation (Epoch change)

When a member is added or removed, the entire ACT is rebuilt with a new `sk_group`, new `content_key`, and incremented `epoch`. Previous keys are invalidated. Messages from epoch N cannot be decrypted with keys from epoch N+1.

---

## Summary

| Operation | Algorithm | Purpose |
|---|---|---|
| Identity & signing | secp256k1 | Prove who sent the message |
| Keypair derivation | SIWE + sha256 | Bridge Ethereum identity to X25519 |
| Key agreement | X25519 (Curve25519) | Derive shared secrets |
| Key derivation | HKDF-SHA256 | Expand shared secrets into distinct keys |
| Encryption | XSalsa20-Poly1305 | Encrypt and authenticate messages |

X25519 is the load-bearing primitive for all confidentiality in Whispery. secp256k1 handles all identity and non-repudiation. Neither is asked to do the other's job.
